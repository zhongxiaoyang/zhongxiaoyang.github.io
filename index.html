理解MongoDB的并发安全

发表于 2020-04-28 | 分类于 数据库相关 ， MongoDB

一般我们处理并发问题都不可避免要考虑性能和数据安全问题，数据安全指的是在并发场景下如何考虑并发写入的安全性和并发读取的一致性问题。本文是对MongoDB内置的并发安全问题的处理机制做个了解，不会涉及业务层面的并发安全问题

摘要

3.0版本之后的MongoDB存储引擎使用了WiredTiger，本文的内容也是基于3.0版本之后的。MongoDB内部是使用锁机制和并发控制措施来确保数据的一致性的，这些机制和措施保证了对单个文档的写入是安全的，并且客户端不会看到不一致的数据。

MongoDB的锁机制

在介绍MongoDB锁机制之前，为了更好的理解，先讲解一下什么是锁机制以及一个普通的操作是怎么使用上锁的。

锁的字面理解就是把门锁上，外面的人在锁打开之前是无法进入里面的。当每个人进入房间都把门锁上，出来的时候把锁打开，就可以保证每次只会有一个人进入房间，其他人只能排队。同样的道理，当一个每次操作某个文档的时候给这个文档加上锁，那么就可以保证同一个文档只能有一个读写操作，不会出现并发写入，这样也就确保了数据写入的并发安全。

我们平时使用的锁分两种类型：共享锁和排它锁。

  • 排它锁：相同的锁不能同时操作，只有一把钥匙，每次只能进入一个人，一般用在写操作上，表示同一条数据不能并发写入
  • 共享锁：相同的锁可以同时操作，有多把钥匙，有相同钥匙的人都可以进入，一般用在读操作上，表示同一条数据允许并发读取

MongoDB的操作如何使用锁机制的呢？

当我们执行一个update操作更新一个文档时，MongoDB会找到这个文档，给这个文档添加排它锁，这样就保证了这个文档在这个update执行完毕之前不允许其他update操作更新这个文档。

MongoDB的锁的类型

锁的级别

MongoDB支持不同粒度级别的锁，包括

  • 全局锁：给实例下所有数据库加锁
  • 数据库级别：给单个数据库加锁
  • 集合级别：给指定集合加锁
  • 文档级别：给指定文档加锁

MongoDB允许存储引擎在集合级别之下实现自己更细粒度的锁，如WiredTiger引擎支持的文档级别锁

锁的类型

MongoDB使用读锁和写锁，读取使用共享锁（S），写入使用排它锁（X），此外还用到了意向锁，如意向共享锁（IS）和意向排它锁（IX）。整体分类情况如下

  • 读锁
    • S：共享锁
    • IS：意向共享锁
  • 写锁
    • X：排它锁
    • IX：意向排它锁

锁的兼容性

锁的兼容性指的是同级别锁的兼容性，MongoDB的不同锁的兼容性如下（其实不限于MongoDB，这种兼容性普遍适用于各种数据库）

IX IS X S
IX 兼容 兼容 冲突 冲突
IS 兼容 兼容 冲突 兼容
X 冲突 冲突 冲突 冲突
S 冲突 兼容 冲突 兼容

MongoDB的加锁机制

MongoDB在给某个粒度级别加锁时，所有高于该级别的层面都使用意向锁。比如对一个文档加X锁，对与该文档对应的集合、数据库等层面会添加IX锁

意向锁

使用意向锁的目的是利用锁与锁之间的互相兼容性来提高并发性能。举个例子说明一下：一个文档D1对应的集合和数据库分别是C1和D1，假设在更新（update）D1的过程中又执行了删除集合C1操作，在有使用意向锁和没有使用意向锁的执行情况将会是

  • 没有意向锁的情况

执行update过程中文档D1上会加X锁，删除集合C1会尝试给集合添加X锁，但是由于两个X锁时不兼容的，所以给C1添加锁之前会先遍历所有文档的加锁情况，最终加锁失败

  • 使用意向锁的情况

执行update过程中会给C1集合和D1数据库加上IX锁，给D1加X锁。执行集合C1删除操作的时候会发现D1和C1添加了IX锁，与X锁不兼容，加锁失败

让渡锁

有些某些的操作，MongoDB不会再整个操作过程中一直占用某个锁，它会在特殊的场景下把锁让渡出来给其它操作使用。这些操作主要包括长时间运行的读写操作（例如查询，更新和删除）在许多条件下都会进行让渡(yield)。MongoDB如果单个修改文档的操作，影响带有multi参数也会让渡锁。

  • 需要长时间执行的事务操作，这类操作会读取大量数据到内存，在这个空隙间可能会把锁让渡出来。当数据准备就绪，操作又会重新获取锁继续操作
  • 读写操作影响多个文档时，可能会把锁让渡出来。比如操作update({}, {}, {multi: true})匹配了多个文档，MongoDB并不会一次性给全部匹配的文档添加X锁，而是每次给当前更新中的文档添加X锁。这样可以做到还没有更新到的文档也可以做其它操作。

正式因为MongoDB的让渡锁机制，所以只能保证单个文档的操作是原子性的，无法保证多个文档的操作也是原子性的。所以当一个操作影响到多个文档时，单靠锁机制是无法满足整个操作的原子性的

集群分片的加锁情况

集群分片功能是MongoDB实现分库分表的机制，集合的数据会分摊到每个分片节点存储。MongoDB在分片集群中，只锁定指定的分片，不会锁定整个集群。

比如集合共有9000条数据，集群中共有3个分片，每个分片存储3000条数据。当用户更新某个分片的某条数据，只会给这个分片的数据库、集合添加IX锁，给文档添加X锁，另外两个分片不加锁。

MongoDB常用操作的并发安全

insert

insert操作是并发安全的，而insertMany操作则由于让渡锁，并发是非安全的。

update和findAndModify

  • 更新单个文档是并发安全的，每个文档更新的时候使用X锁，该锁保证了单个文档不会出现并发更新
  • 更新多个文档(multi)并发是不安全的，因为MongoDB有让渡锁机制。在整个操作执行过程中，让渡了锁的文档接受其他操作，所以并发非安全。但是单个文档更新过程中是并发安全的

同理updateMany操作并发也是非安全的

upsert

upsert操作并发是非安全的，原因在用整个操作的过程是非原子性的。该操作的执行过程为

  • 根据参数条件查询文档，判定upsert条件是否满足，该过程是读取操作，对应文档的S锁。而S锁时是可以并发的
  • 根据查询出来的条件，如果满足就插入新文档，如果不满足就更新文档

并发不安全的因为在于该操作是两个操作合并的操作，有可能导致多个upsert同时操作，并发获取条件的情况

Redis使用笔记之主从备份

发表于 2020-04-25 | 分类于 数据库相关 ， Redis

这里主要介绍Redis通过主从模型实现高可用的机制，并且介绍到哨兵机制的使用

Redis的高可用

单点服务始终会面临数据安全和服务故障问题，Redis针对这个问题提供了主从备份机制，其实很多服务都提供了主从备份机制，如MongoDB。

主从备份机制简单来说就是有多个Redis进程，这些进程一般配置在不同的机器中，其中一个进程充当master节点，其余进程充当slave节点。正常master节点向客户端提供读写服务，slave节点只是跟master连接实现数据备份，当master节点故障或者停止服务，slave中的某个节点会替换master成为新的master节点提供服务，以此实现Redis服务的高可用。

Redis主从服务最合适的节点数为3个，包括1个master节点和2个slave节点

主从服务的数据备份机制

正常主从备份模型我们比较容易理解master节点写入数据，slave节点会自动更新数据。但是Redis的数据是有过期机制的，它的数据同步和过期数据管理机制是怎样的，很有必要了解一下。

参考官网

三种机制

  • 主从节点连接良好
    • master节点想slave节点发送命令流来保存数据同步。简单讲就是master节点的数据有变化有会发送指令给slave节点同步数据，所以slave节点无需管理键过期与否等逻辑
  • 主从节点端口重连
    • 不管因为网络问题或者超时断开连接等原因，当slave节点再次连接master节点后并不会重新全量备份数据，而是备份从断开时间开始的增量命令流
  • 无法进行增量命令流同步的情况
    • 不管什么原因，当slave节点无法使用增量命令流来同步连接之前的旧数据（如slave节点是在中途加入的，或者断开时间过长难以追溯完成的命令流），会进行全量同步。
    • 全量同步过程是master节点根据当前进行RDB持久化，然后将持久化的.rdb文件发送给从服务器。并在向服务器数据恢复过程中继续发送新数据修改的命令流保持数据实时性

异步同步

主从节点同步过程中是异步完成的，也就说说数据同步过程中，Redis主进程是可以继续处理客户端请求的。

在主从节点同步过程中，master节点内部会维护一个队列，用来存放即将发送给从服务器的命令流，另外每个slave节点性能是有差异的，不同slave节点同步的进度又快又慢，master节点则通过记录每个slave节点同步的偏移量来知道不同slave节点同步的进度。

Master节点维护的队列长度是有限的，当偏移量超出队列长度的slave节点的同步就会变成重新同步（全量同步）。偏移量超出队列长度的情况一般有某个slave节点与master节点断开连接比较长时间或者该slave节点是中途加入进来的。

master节点持久化功能

有了主从备份模型，数据的安全性有了一定的保障，当master节点的压力越来越大，性能受到考验的时候，可以考虑关闭master节点的持久化功能来提高master的性能。（完全不建议这样做，因为当master节点重启后仍继续充当master节点的时候，它数据是空的，会导致slave节点同步后数据都为空）

故障转移-哨兵模式

哨兵是Redis内置的机制，安装Redis的时候在 /usr/local/bin 下会生成 redis-sentinel 命令。安装Redis的目录下除了有redis.conf配置文件外，还有sentinel.conf配置文件，专门针对哨兵机制的配置的。

这里主要讲如何通过哨兵机制实现故障转移，也就是当master节点宕机了，slave节点能够挑选出新的master提供服务，以此实现服务的高可用，对于哨兵的其他知识不会深入探讨。

协议介绍

HTTP协议发展至今已经20多年了，至今仍在大量使用，现在已经成为网页开发必须掌握的知识了。

HTTP也叫超文本传输协议，是TCP/IP协议的应用层协议，也是万维网的数据通信基础。该协议的工作流程概括起来也就是：客户端请求和服务端响应，这个工作流程是建立在TCP连接之上的。一个完整的基于HTTP协议的请求响应过程主要包括一下几个流程：

  • 客户端和服务端建立TCP连接
  • 客户端请求服务端
  • 服务端响应客户端
  • 请求完毕，关闭TCP连接

这里可以简单描述一下互联网、万维网和HTTP协议之间的关系。互联网也就是多台计算机连接在一起实现相互之间可以互相访问，而我们平时说的互联网其实就是最大的全球互联网，它把全球的计算机连接在一起。互联网中有很多服务，万维网就是其中一个，它使用URL（统一资源定位器）定位资源，然后采用请求-响应的方式让请求的一方可以访问服务的一方的资源。万维网并不是一个互联网协议，但是它背后有一系列的协议和标准（Web协议族）来支撑着，其中HTTP协议就是其中之一，该协议默认使用80端口。

随着互联网的发展，HTTP协议也发展了好几个版本，但是协议的工作流程本质上没有变化，变化的主要是为了适应互联网的快速发展而做的相关升级。

发展历史

HTTP协议最初在20世纪90年代出现，随后经历了0.9、1.0、1.1和2几个重大版本。其中1.1版本是W3C和IETF协调定义的。后来2015年IETF正式发表了HTTP/2.0标准，正式取代HTTP/1.1的实现标准。

HTTP/0.9

该版本在1991年发布。这个版本现在来看很简单，主要有以下几点：

  • 不传输数据包，只能传输文字，主要规定通信双方的通信格式
  • 没有数据类型（Content-type）概率，请求的内容也只有HTML网页字符串
  • 只有GET请求方法
  • 整个通信过程可以简单概况为
    • 客户端与服务端建立TCP连接
    • 客户端请求服务端获取某个网页信息
    • 服务端依据请求返回网页信息
    • 请求完成，关闭TCP连接

HTTP/1.0

这个版本大约在1995年左右兴起，内容升级了很多，前后也因为版本改动较大花了很长时间来确定和统一标准。这里面也发生了很多有趣的故事，多多少少对该版本的发布产生影响：

  • 1993年美国国家超级计算机应用中心（NCSA）研发了Mosaic浏览器，能显示HTML的图像
  • 1994到1995年，网景通信公司和微软公司纷纷发布自己的浏览器
  • Apache0.2版本出现，HTML也进入2.0版本
  • 1995年左右，微软和网景浏览器大战，各自私下拓展HTML，没有严格按照Web标准化发展，正因为这件事慢慢发展，HTML的兼容问题也慢慢出现
  • 2000年左右，网景公司衰落，Firefox浏览器，兴起，浏览器大战继续

新增的功能

  • 传输内容上不仅仅可以传输文字，还能够传输图像、视频和二进制文件
  • 请求方法除了GET，还有POST和HEAD
  • 请求、响应体加了头部元数据信息，也就是我们熟知的header
    • 比较明显的是请求头部信息添加了版本号、User-Agent、Accept等信息
    • 响应头部信息包含了版本号、Content-type、内容长度等信息。而头部信息和数据实体中间使用一个空行隔开
    • 1.0版本规定头部信息必须是ASCII编码格式，数据则可以是任何格式
  • 状态码
  • 字符集
  • 分段传输
  • 鉴权、缓存、内容编码等

1.0版本的不足之处

这个版本发布不久，性能问题就显现出来了。该版本客户端和服务端的通信过程本质上还是一样的，每次请求建立一次TCP连接。现在来看我们都知道TCP连接是很珍贵的，首先是连接数量有限，然后频繁的建立TCP连接是消耗性能的。简单来说1.0版本的不足主要是性能问题。

这个版本其实也做了一下优化来解决这个问题，比如添加Connection字段，让客户端和服务器端不要关闭TCP连接，以此来实现多次请求公用一个TCP连接。但是因为这不是标准化的解决办法以及不能同时发起多个请求，所以不到半年，就发布了1.1版本。

HTTP/1.1

这个版本进一步完善了HTTP协议，而这个协议也一直被使用至今。这个版本最明显的就是TCP连接默认是不关闭的，除非通信双方一段时间内没有任何通信活动或者其中一方主动关闭连接。除此之外，很多浏览器对同一个域名可以同时建立多个TCP连接。这就极大地提高了通信的效率。

此外，该版本添加了管道机制，一个TCP连接可以同时发起多个请求，也就是说同一个TCP的请求不用一个接一个排队请求，可以多个请求同时进行。

分段传输

  • Content-length

管道机制带来的就是一个TCP连接可以同时发起多个请求，这也就意味着请求的回应也包含多个。1.0版本可以通过TCP连接关闭来确定数据已经全面回应接受，但是1.1版本之后显然这种方式不再适用，因此添加了Content-length字段来解决这个问题。简单来说不管一个请求的响应数据是一次性回应还是分多次回应，客户端都可以根据这个字段来判断数据是否已经全部接受。

  • 流模式：stream

这个版本的传输模式也变了，从缓存模式改成了流模式。

其他功能

  • 请求方法添加了PUT、DELETE等
  • 请求头信息添加了Host字段

服务端响应请求是按顺序的，所以即使客户端请求的顺序，服务端最终也会按顺序一个一个的排队响应。这个就是“队头阻塞”

二进制传输

前面版本每个请求的头信息都是独立的，也就是每次请求都需要发送全部的头信息，该版本对这个进行了优化，授信是头信息要经过压缩，然后重复请求的头信息通过一些机制来避免重复发送

服务端主动向客户端推送

这个版本中服务端可以主动给客户端推送信息，比较容易理解的例子就是有些数据是客户端一定回请求的，所以在客户端请求之前提前给客户端推送过去。比如用户请求了一个网页，跟这个网页相关的一些静态资源就可以构成服务端主动推送的数据了。

请求方法和状态码

请求方法

服务端的资源大体可以分为静态的和动态的资源。静态资源可以理解为URL，动态资源可以理解为API。当然对应前端而言，它不管是URL还是API，他都理解成附带Content-type的URL，只是除了纯粹的获取资源外，还能根据不同的请求方法和请求数据产生交互的效果。这就是请求方法的作用。一般常用的请求方法和作用为

  • GET：获取资源
  • POST：新建资源
  • PUT：修改资源
  • DELETE：删除资源

这是常用的请求方法，其他方法这就不一一列举。旧版本的HTTP协议，新建、修改和删除资源基本都是通过POST方法来实现，不过随着现在RESTful API标准，不同的请求方法应该对应不同的资源处理场景。

状态码

客户端根据URL请求服务端，它可能对应静态资源的路径，也有可能对应一个接口地址，但是不管如何，静态资源也有可能不存在、接口处理也有可能出现异常，所以服务器需要明确告知用户请求处理的结果。状态码就是用来区分不同的请求处理结果的。常用状态码主要分成5大类，基本也是按照状态码的开头数字区分

  • 2xx状态码：处理成功。最常用的是200，表示请求处理成功和响应
  • 3xx状态码：重定向
  • 4xx状态码：客户端错误
    • 400：一般是参数有误
    • 401：未登录授权
    • 403：没有权限，不允许访问
    • 404：资源不存在
    • 405：请求方法不正确
  • 5xx状态码：服务端错误
    • 500：客户端请求没问题，服务端处理异常
    • 503：服务器不能处理请求，如服务器在维护中

Redis使用笔记

Redis

什么是Redis

  • NoSQL数据库
  • key-value存储
  • 分布式缓存中间件，支持水平拓展
  • 提供海量数据存储和访问，数据放在内存中，读取速度很快

为什么用缓存

有时即使数据库做了很多优化：读写分离、分库、分片、集群，但是依然导致数据库慢查询甚至宕机。一个主要的原因就是常规数据库如MySQL、MongoDB等数据存在硬盘中，读取数据性能低下。

很多时候导致数据库问题的操作都是查询操作引起的，这里说的是很多情况下

所以结合这两者，把一些经常访问到的数据缓存起来，当用户访问已缓存的数据就可以直接从缓存中获取数据返回，整个访问过程无需经过数据库操作。

对比Ehcache、Memcache

  • Ehcache
    • 优点：Apache认证、java开发；基于JVM缓存
    • 缺点：单点，不支持集群；不支持分布式拓展
  • Memcache
    • 优点：key-value存储；多核多线程，内存和CPU使用率高
    • 缺点：不能持久化；不能容灾
  • Redis
    • 优点：数据类型丰富；支持持久化：RDB、AOF；支持主从、集群和分布式
    • 缺点：单核，建议多实例充分利用CPU；单线程，不建议大数据量存储和操作

过期键清除策略

Redis使用惰性删除和定期删除两种策略，两种方式配合服务器可以更好地在使用CPU时间和避免浪费内存空间之间取得平衡

  • 惰性删除：放任key过期不管，但是每次操作这个key都检查一下是否过期，如果过期就删除
  • 定期删除：每隔一段时间程序就检查一下数据库，删除里面的过期key

RDB持久化策略不会备份过期键，恢复的时候已过期的键也会被忽略不会载入

AOF持久化策略只要key没有被删除就会做备份，当key被删除后，程序会向AOF文件追加del指令删除该key。重写的时候如果key被删除了，就会移除掉。

Redis是内存数据库，安全性很重要。持久化是解决数据安全问题的其中一个方式。Redis提供了RDB和AOF两种持久化方式，各有不同优缺点，当然其中一个方式的缺点就是另一个方式的优点，使用者可自行选择。

持久化机制

RDB备份方式每次备份都是全量备份，备份过程会生成.rdb文件，备份时生成新的文件，备份完成后移除旧的备份文件，以此保证最新状态都保存在dump.rdb。

也正因为每次备份都是全量备份，当前内存中有多少数据就保存多少数据，所以有很多种方式可以优化备份存储工作，比如使用二进制压缩存储，这种压缩存储方式使得这种备份后的文件相对AOF备份的文件小的多

另外这种备份方式文件存储的都是数据，Redis启动恢复的时候只需要把数据加载到内存中即可，这也是RDB备份方法相对AOF备份方法恢复更快的原因

在RDB备份的时候，Redis服务器可以选择fork一个进程来操作持久化备份工作（bgsave指令），期间不影响服务的正常运行。

Redis有两个指令可以触发该持久化备份，save指令和bgsave指令，两个指令不同之处

  • save：立即备份，服务端主进程执行备份，因为Redis是单线程的，所以备份期间不能处理客户端请求，慎用该指令，特别是数据量大的生产环境
  • bgsave：fork子进程执行备份，服务端（父进程）可以正常处理客户端请求

bgsave指令执行过程中发起的save和bgsave指令都会被拒绝，后面AOF持久化会用到bgrewriteaof指令，三个指令并发关系是

  • bgsave指令执行期间，发起的save指令会被拒绝，发起的bgrewriteaof指令延迟到bgsave完成后执行
  • bgrewriteaof指令执行期间，发起的save和bgsave指令都会被拒绝
    • 这是出于性能考虑，同时fork多个进程会影响Redis服务端的性能

消息队列Beanstalkd使用详解

发表于 2019-08-01 | 分类于 JavaScript ， NodeJs

公司目前用着阿里云的MNS消息队列，考虑到MNS依赖阿里云服务，准备使用beanstalkd替换掉MNS

Beanstalkd

Beanstalkd是一个轻量级、高性能的分布式内存消息队列。

轻量级

  • 安装简单
  • 使用简单，没有复杂的消息队列模式，只提供队列模式，消费者也只能通过主动方式获取新消息
  • API简单，无论是生产者还是消费者，API都很简单。生产者负责消息的发送就完事了；消费者获取消息、处理消息、更新消息状态也完事了。

高性能

高性能源于Beanstalkd是内存消息队列，消息存储在内存中（当然也支持持久化），所以消息的存取都很快。

不足

管理队列

Beanstalkd尚无提供删除一个tube的操作，只能将tube的job依次删除，并让Beanstalkd来自动删除空tube。Beanstalkd的队列是通过自动管理机制完成的。生产者调用use方法指定发送队列、消费者调用watch方法指定消费队列的时候如果队列不存在会创建。当队列消息被消费完，并且所有生产者和消费者端口指定该队列，这个时候队列会被自动清理。

客户端认证

Beanstalkd不支持客户端认证机制，开发者将应用场景定位在局域网。

基本角色

消息队列一般都有几个重要的角色：【消息、队列、生产者、消费者】。只是在Beanstalkd中管消息叫job，管队列叫tube。下文为了方便编辑，还是叫会消息和队列。

生产者负责把消息发送到队列，消费者从队列中拉取数据、消费消息、更新消息状态。

Beanstalkd中提供一个默认的队列（名称为：default），当生产者发送消息的时候如果没有明确指定队列，消息会发送到default队列。

消费者也会模式消费default队列的消息，即使指定了其他队列作为消费队列。如果不想消费default队列的消息，需要使用ignore方法把default队列忽略掉。

消息生命周期

各个状态解析

  • ready - 正常待处理消息，出于这个状态的消息会等待消费者前来消费它
  • delayed - 延时中，生成者发送消息的时候指定了延时时间。延时中的消息不会被消费，过了延时时间，消息会变成ready状态
  • reserved - 消费中，消费中的消息后期可能变成下面几种结果
    • 消费者调用release方法，让消息重回ready或者调用release方法时指定延时参数变成delayed状态
    • 消费者调用deleted方法，删除消息
    • 消费者调用bury方法，让消息隐藏起来
    • 消费者挂了或者显示调用disconnect，与Beanstalkd断开连接，消息重回ready状态
  • deleted - 已删除，消息动队列中删除。其他状态的消息都可以调用删除方法删除消息。
  • buried - 隐藏起来的消息，隐藏的消息有下面几个结果
    • 消费中调用kick方法后，消息重新变回ready状态
    • 被删除，变成deleted状态
    • 处理时间超过ttr值，自动回到ready状态

ttr 消息处理超时

生成者发布消息的时候，会指定ttr（消息处理超时时间），消费者消费消息超过这个时间，消息会自动回到ready状态，重新消费。消费者如果觉得知道时间内不能处理完任务，可以通过touch方法重新计算ttr

机制说明讲的很清楚，首先unionid是基于微信开发平台的，一个开发平台账号下的多个小程序、网站应用和公众号应用共享一个unionid。比如A用户使用了小程序1、小程序2、公众号1，这三个应用分别有不同的openid，但是当把这三个应用绑定到同一个开发平台他们都有同样的unionid，我们可以利用unionid来标志不同的应用是否属于同一个人。

经常会有这种需求，比方说我们有一个小程序，后期因为业务太多，将其中的部分业务移到新的小程序，这样新旧小程序的openid不一致可能导致数据的复用上出现麻烦，这个时候根据unionid可以轻松实现数据复用或者迁移。

unionid获取渠道

绑定了开发者帐号的小程序，可以通过以下途径获取 UnionID。

  1. 调用接口 wx.getUserInfo，从解密数据中获取 UnionID。注意本接口需要用户授权，请开发者妥善处理用户拒绝授权后的情况。
  2. 如果开发者帐号下存在同主体的公众号，并且该用户已经关注了该公众号。开发者可以直接通过 wx.login + code2Session获取到该用户 UnionID，无须用户再次授权。
  3. 如果开发者帐号下存在同主体的公众号或移动应用，并且该用户已经授权登录过该公众号或移动应用。开发者也可以直接通过 wx.login获取到该用户 UnionID ，无须用户再次授权。
  4. 用户在小程序（暂不支持小游戏）中支付完成后，开发者可以直接通过getPaidUnionId接口获取该用户的 UnionID，无需用户授权。注意：本接口仅在用户支付完成后的5分钟内有效，请开发者妥善处理。
  5. 小程序端调用云函数时，如果开发者帐号下存在同主体的公众号，并且该用户已经关注了该公众号，可在云函数中通过 cloud.getWXContext 获取 UnionID。
  6. 小程序端调用云函数时，如果开发者帐号下存在同主体的公众号或移动应用，并且该用户已经授权登录过该公众号或移动应用，也可在云函数中通过 cloud.getWXContext 获取 UnionID。

消息确认

RabbitMq 怎么决定一个消息已经被正常处理了呢？这就是消息声明周期的最后一步：消息确认。消息一旦被确认，RabbitMq就会知道这个消息已经被正常处理，然后将消息从队列移除。

消息分发

当我们有多个消费者在消费统一个队列的时候，消息是怎么分配给众多消费者的？

首先需要说明的一点就是，RabbitMq的一条队列的一个消息只会发送给一个消费者处理，永远不会将同一个消息发送给多个消费者同时处理。只有一种情况一条消息前后经历过多个消费者处理，那就是前面处理的消费者没有确认消息，在它断开连接之后这条消息重新变成待消费状态之后发送给另外一个消费者处理。RabbitMq拒绝这种会导致功能变得复杂且难以理解的功能，如果我们期待一条消息可以被多个消费者处理，那我们应该声明一个fanout交换器，每个需要处理消息的消费者都将队列绑定到这个交换器，那么当有消息就可以收到了。

Round-robin（轮询分发）

默认这种方式，RabbitMq会将消息分摊给每个消费者。如果当前队列有10条消息，有两个消费者，则[1, 3, 5, 7, 9]条消息将推送给先监听队列的消费者，[2, 4, 6, 8, 10]条消息将推送给后监听队列的消费者

但是有时候我们不希望使用默认的这种方式，毕竟两个消费者处理能力不一样，处理消息能力强的消费者可能很快处理完了5条消息，处理能力弱的能需要很长的时间来处理一条消息。这个时候我们就希望RabbitMq按照处理能力分发消息，这就是下面讲到的另一个分发规则

Fair dispatch（公平分发）

这种方式，消费者消息的时候不能将 noAck 参数设置为true，消息确认方式应该保证为手动确认

这种方式，我们使用basicQos( prefetchCount)方法，来限制RabbitMQ只发不超过指定条数的消息给同一个消费者。当消息处理完毕后，有了反馈，才会进行第二次发送。

下面的逻辑将prefetch设置为1，表示消费者处理消息的并发值为1。RabbitMq会给消费者发送1条消息，在消息被确认之前不会发送第二条消息，而一旦消息被确认，马上发送一条新的消息，以保证该消费者始终有一条处理中的消息。

代码的不同之处在于加了代码：channel.prefetch(1)。这行代码的位置没有限制，它对信道做了限制，一般在channel.consume之前指定就可以了。

如果不控制并发prefetch，或者noAck为true，RabbitMq会一次性将所有消息分摊给消费者，这经常会出现问题，如果两个消费者，队列有一百条消息，则每个消费者各得50条消息，相当于并发50。这种方法对服务不利，我们不应该出现这种情况，应该控制并发保证服务高可用。

  • queue1的绑定键为log.*，所以他能匹配到一个包含两个部分，且第一个部分为log的路由键。它接收了代码中第 [1, 2, 3, 7, 8, 9] 条消息
  • queue2接收了第 [4, 5, 6] 条消息
  • queue3接收了第 [1, 4] 条消息
  • queue4的绑定键为log.#，所以它能匹配到第一部分为log的不限部分数的路由键。它接收了第 [1, 2, 3, 7, 8, 9, 10, 11] 条消息
  • queue5接收了第 [4, 5, 6] 条消息
  • queue6接收了第 [1, 4] 条消息
  • queue7接收了每一条消息

队列绑定交换器

绑定交换器的部分在前一部分已经讲了很多，关于绑定的概念也有了些许了解。但是当我们刚安装启动RabbitMq的时候，我们从management界面上的exchanges导航下可以看到，其实RabbitMq已经默认为我们提供了一些交换器，平时我们不一定每次都需要自己声明新的交换器，拿已经存在的交换器就好了。

首先我们可以看到，这些交换器都是持久化的(在features里面有个青色的D)，这里包含了2个direct类型、2个headers类型、1个fanout类型以及3个topic类型的交换器

第一个direct类型的交换器是RabbitMq默认的交换器，其他的交换器也只是RabbitMq提供好方便我们使用而已

默认交换器点进去不会列出已经绑定的队列列表，它是所以队列默认会绑定的交换器。也就是每个队列声明成功之后都会绑定到这个交换器，绑定键就是队列名称，客户端不能显示使用这个默认交换器绑定或者解绑队列，这个交换器也不能删除。

默认交换器如何使用：

当我们使用sendToQueue发送消息给某个队列的时候，默认交换器就用上了。消息到达默认交换器，默认交换器根据队列名字将消息路由到指定队列

所以我们可以发现，所有的消息都是经过了交换器，最终才到达队列的

  • API后台 录入一个咨询任务到 数据库，并返回给 客户端 任务录入成功

  • 后台根据自己的能力，每次处理一定量的任务，不会因为突然到来的大量请求把服务搞垮

  • API后台 收到 客户端 的 HTTP 请求后发送一条消息到 MQ，然后返回给 客户端 请求成功

  • 业务使用了异步，服务端不需要并发处理所有用户的请求，不会给服务端太大的压力
  • 解耦
    • 用户不需要在页面中静静地等待，他大可以在请求完之后去做其他的事情，等恰当的时机看短信就好
    • API后台 不需要直接调用 核心服务，实现了服务之间的松耦合
  • 控制并发，核心服务 可以设置并发的量(prefetch)，MQ 会根据这个值控制一次性推送消息的量给 核心服务

MQ 架构图

这一节会讲到在 MQ 的整个架构中，我们的服务是什么角色，MQ 作为什么角色存在，他们是如何结合在一个工作的，我们先看一张从网络上爬下来的 MQ 的简单架构图：

图2.2.1

图2.2.1 中虽然没有一个中文看着有点晕，但是其实挺容易理解的。上一节 股票咨询功能 的例子中我们知道有一个 消息的发送方(API后台)、消息的接收方(MQ) 以及 消息的处理方(核心服务)，再看下这个张图，也能猜得出 P 就是消息的发送方(生产者)，C 就是消息的处理方(消费者)，中间的部分就是 MQ服务器 了。

在 MQ 的整个流程当中，会有两类角色：客户端(生产者和消费者)、服务端(MQ)。客户端需要与服务端建立连接才能发送或者消费消息，这个机制看起来服务端有点像一个数据库，只是他的数据存储在内存当中。生产者和消费者的下面会详细介绍，它们都是对于使用 MQ 来说必不可少的角色

MQ服务器 内部有两个主要的部件：交换器(exchange) 和 队列(queue)。交换器就好比如 门，生产者发送的消息首先经过交换器这扇大门，然后经由交换器找到对应的队列后将消息列入队列，交换器有一种机制可能知道这个消息应该放到哪个队列里面；队列就是存储消息的地方，先进来的消息会优先被消费，这是队列先入先出的特点。

说了前面这几段话，生产者发送的消息首先经过 MQ服务器 的交换器，经由交换器将消息放到指定的队列。这里的消息就是数据，生产者发送消息就是将一个数据(可以理解成请求参数，可以是字符串这些基本类型，也可以是对象、数组甚至二进制类型)发送给MQ服务器，MQ服务器在将这个数据发送给消费者，消费者接受到这个消息就可以看到消息的数据了。

生产者(Produce)

生产者负责发送消息，一个生产者可以将消息发送到多个消息队列里面，具体发送到什么队列，只需在发送之前执行消息发送的队列名称就行了，使用很简单。生产者整个流程可以总结为：

  • 连接 MQ 服务器，与服务器建立一个 TCP 连接
  • 获取信道
  • 声明交换器
  • 声明队列
  • 创建消息
  • 发布消息
  • 关闭信道
  • 关闭连接

消费者(Consume) - 负责消费 MQ服务器 的消息

消费者负责消费消息，一个消费者可以消费多条队列的消息，多个消费者也可以消费同一个队列的消息。消费者消费消息的流程可以总结为：

  • 连接 MQ服务器，与服务器建立 TCP 连接
  • 获取信道
  • 声明交换器
  • 声明队列
  • 把队列跟交换器绑定起来
  • 消费消息
  • 关闭信道
  • 关闭连接

交换器

生成者和消费者的流程都可以声明交换器，交换器也是图中绿色的部分，交换器配置了一个消息路由到队列的规则，RabbitMq提供了4中交换器

  • direct交换器：严格匹配
  • fanout交换器：将收到的消息广播到绑定的队列上
  • topic交换器：可以将不同源头的消息到达同一个队列

MQ的历史

来自书籍《RabbitMq实战第一章节》

在1983年，一个年轻的工程师提出一个问题：为什么没有一种通用的通信系统，可以解决应用程序建繁重的信息通信工作？

来自MIT的硬件设计教育工作者Vivek Ranadive设想了一种通用的软件总线，就像主板上的总线那样，供其他应用程序接入，因此在1983年Teknekron诞生了。

1985年Ranadive找到了他的第一个客户以及软件总线天生用来解决的问题：金融交易。但是交易员的小隔间内挤满了用来完成交易的不同终端，每台终端上显示着不同类型的信息。Ranadive发觉这会是个机会：替换掉所有那些终端和孤立的应用程序。在交易员位置上将会是Ranadive的软件总线，而桌上只会剩下一台工作站。工作站的显示程序可以作为消费者接入Teknekron软件总线，并允许交易员“订阅”他想要看到的信息。于是发布订阅模式（PubSub）诞生了，同时还诞生了世界上第一个现代消息队列软件：Teknekron和The information bus(TIB)

用不了多久，这种数据传输模型就找到了更多啥手机用途。最终，发布数据的应用和消费数据的应用再也不用直接连接在一起了。他们甚至都不需要对方的存在。Teknekron的TIB允许应用开发者建立一系列的规则去描述消息内容。只要消息按照这些规则发布出去，任何消费者应用都能订阅感兴趣的消息。现在信息的生产者和消费者之间可以完全解耦，并且可以在传输过程中灵活混合。PubSub模型的任意一边可以完全互换不会影响另一边。唯一需要保持稳定的是TIB软件以及标记和路由信息规则。由于在哪个时期金融交易行业门庭若市，因此TIB迅速传播开来。它引起了电信特别是新闻机构的注意。这些行业也特别需要将信息及时发布给千变万化的客户，这也就是为啥1994年大型新闻机构路透社收购了Teknekron。

20世纪80年代，IBM开始研究开发自己的消息队列软件，运用他们在开发DB2时信息投递的丰富经验，开发工作始于1990年，开发3年后IBM MQ产品系列面世。之后17年，MQ系列进化成了WebSphere MQ并统治着商业消息队列平台市场。在那段时间，Ranadive的TIB并没有消失在路透社腹中。相反它任然是企业通信市场的主要参与者，并且更名为Rendezvous而走向繁荣。在1997年Teknekron以TIBCO的形式作为一家独立公司再度出现。童年微软也在消息通信市场崭露头角：微软消息队列(MSMQ)

通过了一系列的革新，消息队列软件主要留住了大型组织机构，他们需要可靠性、解耦以及实时消息通信。为什么MQ不去寻找更大的市场呢？它是如何度过20世纪90年代末网络泡沫的呢？毕竟从Twitter到Salesforce.com，当今所有的这些企业都在努力创建内部方案来解决25年前TIB就已经解决的PubSub问题。一句话：供应商壁垒。商业MQ供应商想要解决应用互通的问题，而不是去创建标准接口来允许不同的MQ产品互通，或者允许应用程序来更改MQ平台。供应商壁垒维持着足够高的价格和利润率，并使得这些商业MQ软件对那些当今繁荣昌盛的初创公司和Web 2.0公司来说遥不可及

结果，中小技术公司并不是唯一一个对高价MQ供应商感到不满的，那些造就MQ产业的金融服务公司对此也激动不已。越是大型金融公司余越不可避免地使用来自众多供应商的MQ产品，来服务企业内部的不同应用。如果应用已经订阅了TIBCO MQ信息，若突然需要消费来自IBM MQ的消息，则实现起来会非常困难。这些产品使用了不同的API，不同的协议，因而毫无疑问无法联合起来组成单一的总线。为了解决这个问题，JMS在2001年诞生了。JMS试图通过公共Java API的方式，隐藏单独MQ产品供应商提供的实际接口，从而跨越了壁垒和解决了互通的问题。从技术上讲，Java应用程序只需针对JMS API编程，选择合适的MQ驱动即可。JMS会打理好其他部分。问题是你在尝试使用单独标准化接口来胶合众多不同的接口，这就像是把不同类型的衣服粘合在一起。使用JMS的应用程序会变得更加脆弱，我们需要新的消息通讯标准化方案

AMQP

2004年，JPMorgan Chase需要一个更好的消息通信解决方案，并开始和iMatix公司一起合作开发Advanced Message Queuing Protocol (AMQP，高级消息队列协议)。AMQP从一开始就设计为开放标准，已解决众多的消息队列需求和拓扑结构问题。凭借开发，任何人都可以执行这一标准，针对标准编码的任何人都可以和任意AMQP供应商提供的MQ服务器进行交互

在很多方面，AMQP承诺把我们的厂商解救出来，并且实现Ranadive最初的愿景：从任何发布者到任何感兴趣的消费者之间的信息，通过一条软件总线实时动态连接起来

RabbitMq

本世纪出，Alexis Richardson创办了专门研究Java对象缓存的Metalogic公司，Matthias在LShift工作，LShift正投入和一家大型软件供应商的合同，专注于语言建模和分布式计算。这些领域方面的北京出发了Matthias对Erlang的兴趣。Erlang是Ericsson原本为了电话交换机而开发的编程语言。引起来Matthias的注意的是Erlang在分布式编程和健壮的故障恢复方面的表现出色

在2005年左右，Alexis创办了CohesiveFT。他们提供应用栈和工具，在今天主键成为云计算。对Alexis来说，应用栈最关键的部分是分布式消息通信。他开始找Matthias探讨AMQP。他们聚集在三个问题上：

  • 那些大型金融机构是否会在意他们的消息代理服务器使用Erlang语言编写的
  • Erlang语言是否真的是编写AMQP服务器的最佳选择
  • 如果采用Erlang语言编写，是否会影响开源社区对它的采纳

这三个问题后面都得到解决，Alexis和Matthias说服了CohesiveFT和LShift共同支持这个项目。他们首先是签约了Matthew Sackman，让他用Erlang语言写了一个原型测试网络延迟。他们很快发现用Erlang编写的分布式计算库有着可原生Socket一样的延迟。因此在2006年，Rabbit Technologies诞生了：一家由CohesiveFT和LShift合资的企业，其拥有者RabbitMq的知识产权

就在当时，AMQP规范的第一份公开草案也公之于众了，作为一份新的规范，AMQP正快速修订。这正是Erlang可以施展拳脚的地方。通过使用Erlang，RabbitMq可以快速开发并跟上AMQP标准前进的节奏。

最初，RabbitMq实现了AMQP的一个关键特性，使其有别于TIBCO和IBM：使用协议本身就可以对象队列和交换器这样的资源进行配置。对于商业供应商来说，资源配置需要通过特定的管理终端的特定工具才可以完成。RabbitMq的资源配置能力使其成为构件分布式应用的最完美的通信总线，特别有助于充分利用基于云的资源和快速开发

就这样一直到今天，RabbitMq广泛使用在小到硅谷的初创公司达到互联网巨头。对于RabbitMq来说，那也许就是最好的事情了。RabbitMq实现了Ranadive的愿景，RabbitMq为我们提供了一个强大的工具来解决那些问题，并给我们带来了一段丰富的消息通信历史，以及适合所有人的可插拔的信息总线

今天，RabbitMq并不是开发消息通信的唯一选择，想ActiveMQ、ZeroMQ和Apache Qpid都提供了不同的开源消息队列方案。

从看消息队列的历史，我们可以知道在漫长的发展中，消息队列是为什么存在的，它的初衷以及发展到现在的功能，它能解决什么问题，能解决我们的什么痛点。参照历史去了解消息队列，会了解的更加深刻一些。

很明显，我们知道了，AMQP是一种规范，它允许来自不同供应商的消息生产者和消费者实现真正的互操作拓展，就如同 SMTP，HTTP，FTP等协议那样。而RabbitMq是完成遵循AMQP规范的，它比竞争对手更加可靠，更能防止崩溃，另外它极易安装和使用。

什么时候需要 MQ

通常，MQ 常用的场景有业务解耦，最终一致性，广播，错峰流控

异步

异步是相对于同步而言的，同步是阻塞，异步是非阻塞。异步就是A调用B之后不需要等待B返回结果而继续执行B之后的代码。

消息队列的实现方法就是异步，生成者发送消息之后不需要等待消费者处理完消息才往下执行，大大提高了请求性能

解耦

使用 MQ 经常可以解决我们业务中系统之间或者模块之间的耦合，很多时候某个系统在处理某个业务的时候依赖其他服务，而一旦形成了依赖关系，则其他系统的稳定性就直接影响了这个系统的稳定性

假设现在有订单管理和积分管理两个系统，对于用户下单请求创建订单的流程

传统的做法是用户下单的时候 订单管理系统 需要调用 积分管理系统 扣减用户积分成果后返回下单成功，如果调用 积分管理系统 失败，订单创建失败

使用 MQ 之后变成用户下单的时候 订单系统 发送消息给 MQ，然后返回下单成功，MQ 在 积分管理系统 正常处理的时候讲消费推送给它处理，这种使用了 MQ 解耦了 订单管理系统 与 积分管理系统 有以下好处

  • 就算积分管理系统出现暂时的无法访问，也不会影响用户正常的下单
  • 订单管理系统处理时间缩短，减少了用户下单等待时间
  • 就算积分管理系统突然改了域名，也不会导致订单管理系统不能请求

就算订单管理系统和积分管理系统不是在两个独立的系统里面，而是一个系统的两个模块，使用消息队列将两个模块解耦也是有意义的。

比如 node.js 中，模块调用我们可能会想到使用 EventEmitter 来解耦模块之间调用，而 EventEmitter 的底层实现方式就是消息队列

最终一致性

最终一致性要求请求能够正常被处理(高可靠)，不会因为某些原因(服务异常、超时)导致请求被异常处理。可以依靠 MQ 来做到最终一致性，因为 MQ 会将消息发送给已经准备好消费的消费者系统处理，在消息处理过程中依赖消费者确认消息来保证消息被正常处理

广播

消息队列的基本功能之一是进行广播。如果没有消息队列，每当一个新的业务方接入，我们都要联调一次新接口。有了消息队列，我们只需要关心消息是否送达了队列，至于谁希望订阅，是下游的事情，无疑极大地减少了开发和联调的工作量。

比如我们有个 用户信息管理系统，包含用户的 注册，认证，更新信息 等功能，另外有 风控，客服，推广等系统 都需要实时知道用户注册了，认证了，改资料了。

传统的做法每当用户注册、认证或者更新信息，我们都需要显示调用其他需要实时知道情况的系统。这就僵硬了，如果这些系统下线了，或者域名变更又或者有新的系统加入通知，这个 用户信息管理系统 都需要做出相应的改动，更恶心的是他们的重构可能要求你配合他们测试或者上线。

使用 MQ 整个过程实现解耦，用户注册的时候 用户信息管理系统 只需关心消息是否发送成功，至于其他需要关心用户注册、认证和更改信息的系统，使用topic监听和消费队列消息就完事了。

广播大家都熟悉，一个广播发布者，多个监听者。就比如订阅报纸，你订阅了南方日报，每当有新报纸，你都能收到一样，你不用管是哪个报社发出来的，或者是哪个送报人员送的，你这需要管你收到没有。MQ 专门为了这类需求提供了topic这种交换器

错峰与流控

前面有说道 消费者 消费消息的时候可以设置消息的并发值，一旦设置了相应的并发值，MQ 会根据消息确认情况确保消费者系统同一时间在处理一定数量的请求，所以这种情况就可以有效避免突然增加的请求导致服务高负荷

试想上下游对于事情的处理能力是不同的。比如，Web前端每秒承受上千万的请求，并不是什么神奇的事情，只需要加多一点机器，再搭建一些LVS负载均衡设备和Nginx等即可。但数据库的处理能力却十分有限，即使使用SSD加分库分表，单机的处理能力仍然在万级。由于成本的考虑，我们不能奢求数据库的机器数量追上前端。

这种问题同样存在于系统和系统之间，如短信系统可能由于短板效应，速度卡在网关上（每秒几百次请求），跟前端的并发量不是一个数量级。但用户晚上个半分钟左右收到短信，一般是不会有太大问题的。如果没有消息队列，两个系统之间通过协商、滑动窗口等复杂的方案也不是说不能实现。但系统复杂性指数级增长，势必在上游或者下游做存储，并且要处理定时、拥塞等一系列问题。而且每当有处理能力有差距的时候，都需要单独开发一套逻辑来维护这套逻辑。所以，利用中间系统转储两个系统的通信内容，并在下游系统有能力处理这些消息的时候，再处理这些消息，是一套相对较通用的方式。

总而言之，消息队列不是万能的。对于需要强事务保证而且延迟敏感的，RPC是优于消息队列的。

对于一些无关痛痒，或者对于别人非常重要但是对于自己不是那么关心的事情，可以利用消息队列去做。

支持最终一致性的消息队列，能够用来处理延迟不那么敏感的“分布式事务”场景，而且相对于笨重的分布式事务，可能是更优的处理方式。

当上下游系统处理能力存在差距的时候，利用消息队列做一个通用的“漏斗”。在下游有能力处理的时候，再进行分发。

如果下游有很多系统关心你的系统发出的通知的时候，果断地使用消息队列吧。

我叫戴宝，2010年12月从湖北调来仲恺农业工程学院；2011年1月至2014年9月在轻工食品学院，任驻白云校区的辅导员；2014年9月至2015年4月9日调自动化学院，依然是驻白云校区的辅导员。2015年4月9日上午10:15分前后，我接到命令，让我从即日起兼任计算科学学院辅导员，马上报到，并且依然驻白云。

在会上，草业12级的刘志超同学开篇就讲：我们不要讲法律，我们不要讲纪律，因为我们认为，法律与纪律下的人生是灰暗的。我们讲一讲人情。可能你们自己也觉得自己的做法已经为法律和秩序所不容，谈起法律来有些心虚，所以才有这么别开生面的开场白。我倒是想请问一下，当你们被抢、被骗的时候，你们为什么要讲法律呢？当你们要求权利的时候，你们为什么要谈与法律天然相生的民主呢？你们需要法律保护时就讲法律，当你们觉得法律限制了自己时，就讲人情，你们不觉得自己有些荒唐和无耻吗？诚然我承认，法律与纪律下的人生可能是“灰暗”的，但没有法律与纪律的人生，必然是没有安全和保障的。萨达姆时代的伊拉克人总是抱怨自己被管的太死，于是美国人说可以让他们得到解放，现在，你们天天都可以从新闻上看到被美国解放了的伊拉克人民的生活：他们不被管得太死的代价就是到处是枪炮和炸弹。既然需要法律的保护，那就必然需要接受法律的限制。因为法律对你个人的限制恰恰是对他人或集体的保护；就像曾经在保护你时、也必然对他人或集体进行了限制一样。从法理上讲，现代法律对“此”的保护是只能通过对“彼”的限制来实现的。所以，现代政治意义上的“自由”，是可以做法律所允许的一切事情，而不是做自己想做的一切事情。1917年4月，美国站在英法一方参加第一次世界大战，威尔逊总统在参战演说中警告德国说：“如果有人认为，只有在对他们有利时才会遵守条约；我就不得不使局势变得，只有遵守条约才对他们有利。”我希望，那些企图讲人情的同学，仔细琢磨一下这位美国总统的讲话，美国不正是你们所欣赏的民主国家吗？

nock 可以做 HTTP 的 mock。单元测试经常会用到它，比如我们写了一个对接的 API，在 API 里面需要调用另一个系统的接口。我们在写这个对接 API 的单元测试的时候没必要每次都发起 HTTP 请求，而是自己模拟这个请求返回数据，以保证单元测试可以不依赖网络或其他系统。

不理想的单元测试

我们应该能够意识到，当我们写的单元测试出现以下问题的时候，其实是不理想的：

  • 连不上网络就跑不通测试
  • 测试很慢
  • 测试跑出来的结果不靠谱，有时通过，有时不通过
  • 测试 case 不足或者测试用例不足以说明代码逻辑准确性，导致就算能够跑得通测试也不能保证逻辑正确、代码设计没有问题
