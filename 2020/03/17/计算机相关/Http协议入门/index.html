了解Http协议

发表于 2020-03-17 | 分类于 计算机相关

协议介绍

HTTP协议发展至今已经20多年了，至今仍在大量使用，现在已经成为网页开发必须掌握的知识了。

HTTP也叫超文本传输协议，是TCP/IP协议的应用层协议，也是万维网的数据通信基础。该协议的工作流程概括起来也就是：客户端请求和服务端响应，这个工作流程是建立在TCP连接之上的。一个完整的基于HTTP协议的请求响应过程主要包括一下几个流程：

  • 客户端和服务端建立TCP连接
  • 客户端请求服务端
  • 服务端响应客户端
  • 请求完毕，关闭TCP连接

这里可以简单描述一下互联网、万维网和HTTP协议之间的关系。互联网也就是多台计算机连接在一起实现相互之间可以互相访问，而我们平时说的互联网其实就是最大的全球互联网，它把全球的计算机连接在一起。互联网中有很多服务，万维网就是其中一个，它使用URL（统一资源定位器）定位资源，然后采用请求-响应的方式让请求的一方可以访问服务的一方的资源。万维网并不是一个互联网协议，但是它背后有一系列的协议和标准（Web协议族）来支撑着，其中HTTP协议就是其中之一，该协议默认使用80端口。

随着互联网的发展，HTTP协议也发展了好几个版本，但是协议的工作流程本质上没有变化，变化的主要是为了适应互联网的快速发展而做的相关升级。

发展历史

HTTP协议最初在20世纪90年代出现，随后经历了0.9、1.0、1.1和2几个重大版本。其中1.1版本是W3C和IETF协调定义的。后来2015年IETF正式发表了HTTP/2.0标准，正式取代HTTP/1.1的实现标准。

HTTP/0.9

该版本在1991年发布。这个版本现在来看很简单，主要有以下几点：

  • 不传输数据包，只能传输文字，主要规定通信双方的通信格式
  • 没有数据类型（Content-type）概率，请求的内容也只有HTML网页字符串
  • 只有GET请求方法
  • 整个通信过程可以简单概况为
    • 客户端与服务端建立TCP连接
    • 客户端请求服务端获取某个网页信息
    • 服务端依据请求返回网页信息
    • 请求完成，关闭TCP连接

HTTP/1.0

这个版本大约在1995年左右兴起，内容升级了很多，前后也因为版本改动较大花了很长时间来确定和统一标准。这里面也发生了很多有趣的故事，多多少少对该版本的发布产生影响：

  • 1993年美国国家超级计算机应用中心（NCSA）研发了Mosaic浏览器，能显示HTML的图像
  • 1994到1995年，网景通信公司和微软公司纷纷发布自己的浏览器
  • Apache0.2版本出现，HTML也进入2.0版本
  • 1995年左右，微软和网景浏览器大战，各自私下拓展HTML，没有严格按照Web标准化发展，正因为这件事慢慢发展，HTML的兼容问题也慢慢出现
  • 2000年左右，网景公司衰落，Firefox浏览器，兴起，浏览器大战继续

新增的功能

  • 传输内容上不仅仅可以传输文字，还能够传输图像、视频和二进制文件
  • 请求方法除了GET，还有POST和HEAD
  • 请求、响应体加了头部元数据信息，也就是我们熟知的header
    • 比较明显的是请求头部信息添加了版本号、User-Agent、Accept等信息
    • 响应头部信息包含了版本号、Content-type、内容长度等信息。而头部信息和数据实体中间使用一个空行隔开
    • 1.0版本规定头部信息必须是ASCII编码格式，数据则可以是任何格式
  • 状态码
  • 字符集
  • 分段传输
  • 鉴权、缓存、内容编码等

1.0版本的不足之处

这个版本发布不久，性能问题就显现出来了。该版本客户端和服务端的通信过程本质上还是一样的，每次请求建立一次TCP连接。现在来看我们都知道TCP连接是很珍贵的，首先是连接数量有限，然后频繁的建立TCP连接是消耗性能的。简单来说1.0版本的不足主要是性能问题。

这个版本其实也做了一下优化来解决这个问题，比如添加Connection字段，让客户端和服务器端不要关闭TCP连接，以此来实现多次请求公用一个TCP连接。但是因为这不是标准化的解决办法以及不能同时发起多个请求，所以不到半年，就发布了1.1版本。

HTTP/1.1

这个版本进一步完善了HTTP协议，而这个协议也一直被使用至今。这个版本最明显的就是TCP连接默认是不关闭的，除非通信双方一段时间内没有任何通信活动或者其中一方主动关闭连接。除此之外，很多浏览器对同一个域名可以同时建立多个TCP连接。这就极大地提高了通信的效率。

此外，该版本添加了管道机制，一个TCP连接可以同时发起多个请求，也就是说同一个TCP的请求不用一个接一个排队请求，可以多个请求同时进行。

分段传输

  • Content-length

管道机制带来的就是一个TCP连接可以同时发起多个请求，这也就意味着请求的回应也包含多个。1.0版本可以通过TCP连接关闭来确定数据已经全面回应接受，但是1.1版本之后显然这种方式不再适用，因此添加了Content-length字段来解决这个问题。简单来说不管一个请求的响应数据是一次性回应还是分多次回应，客户端都可以根据这个字段来判断数据是否已经全部接受。

  • 流模式：stream

这个版本的传输模式也变了，从缓存模式改成了流模式。

其他功能

  • 请求方法添加了PUT、DELETE等
  • 请求头信息添加了Host字段

1.1版本的不足指出

服务端响应请求是按顺序的，所以即使客户端请求的顺序，服务端最终也会按顺序一个一个的排队响应。这个就是“队头阻塞”

HTTP/2

二进制传输

不管是头信息还是数据体，都是二进制传输，也叫数据帧。数据帧包括头部帧和数据帧。

多路复用

请求和响应都可以同时多个进行，不需要根据请求顺序一个个响应

头信息压缩

前面版本每个请求的头信息都是独立的，也就是每次请求都需要发送全部的头信息，该版本对这个进行了优化，授信是头信息要经过压缩，然后重复请求的头信息通过一些机制来避免重复发送

服务端主动向客户端推送

这个版本中服务端可以主动给客户端推送信息，比较容易理解的例子就是有些数据是客户端一定回请求的，所以在客户端请求之前提前给客户端推送过去。比如用户请求了一个网页，跟这个网页相关的一些静态资源就可以构成服务端主动推送的数据了。

请求方法和状态码

请求方法

服务端的资源大体可以分为静态的和动态的资源。静态资源可以理解为URL，动态资源可以理解为API。当然对应前端而言，它不管是URL还是API，他都理解成附带Content-type的URL，只是除了纯粹的获取资源外，还能根据不同的请求方法和请求数据产生交互的效果。这就是请求方法的作用。一般常用的请求方法和作用为

  • GET：获取资源
  • POST：新建资源
  • PUT：修改资源
  • DELETE：删除资源

这是常用的请求方法，其他方法这就不一一列举。旧版本的HTTP协议，新建、修改和删除资源基本都是通过POST方法来实现，不过随着现在RESTful API标准，不同的请求方法应该对应不同的资源处理场景。

状态码

客户端根据URL请求服务端，它可能对应静态资源的路径，也有可能对应一个接口地址，但是不管如何，静态资源也有可能不存在、接口处理也有可能出现异常，所以服务器需要明确告知用户请求处理的结果。状态码就是用来区分不同的请求处理结果的。常用状态码主要分成5大类，基本也是按照状态码的开头数字区分

  • 2xx状态码：处理成功。最常用的是200，表示请求处理成功和响应
  • 3xx状态码：重定向
  • 4xx状态码：客户端错误
    • 400：一般是参数有误
    • 401：未登录授权
    • 403：没有权限，不允许访问
    • 404：资源不存在
    • 405：请求方法不正确
  • 5xx状态码：服务端错误
    • 500：客户端请求没问题，服务端处理异常
    • 503：服务器不能处理请求，如服务器在维护中
