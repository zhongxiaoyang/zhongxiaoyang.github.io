Redis使用笔记

发表于 2020-02-23 | 分类于 数据库相关 ， Redis

Redis

什么是Redis

  • NoSQL数据库
  • key-value存储
  • 分布式缓存中间件，支持水平拓展
  • 提供海量数据存储和访问，数据放在内存中，读取速度很快

为什么用缓存

有时即使数据库做了很多优化：读写分离、分库、分片、集群，但是依然导致数据库慢查询甚至宕机。一个主要的原因就是常规数据库如MySQL、MongoDB等数据存在硬盘中，读取数据性能低下。

很多时候导致数据库问题的操作都是查询操作引起的，这里说的是很多情况下

所以结合这两者，把一些经常访问到的数据缓存起来，当用户访问已缓存的数据就可以直接从缓存中获取数据返回，整个访问过程无需经过数据库操作。

对比Ehcache、Memcache

  • Ehcache
    • 优点：Apache认证、java开发；基于JVM缓存
    • 缺点：单点，不支持集群；不支持分布式拓展
  • Memcache
    • 优点：key-value存储；多核多线程，内存和CPU使用率高
    • 缺点：不能持久化；不能容灾
  • Redis
    • 优点：数据类型丰富；支持持久化：RDB、AOF；支持主从、集群和分布式
    • 缺点：单核，建议多实例充分利用CPU；单线程，不建议大数据量存储和操作

Redis数据类型

操作key

Redis底层是用hash存储数据，也就是每个数据对应一个key-value，因此就有了可以操作不同类型的key的统一指令，这些指令就包括（常用的）

string

Redis 是用C语言开发的，但是字符串实现上没有完全使用C语言的字符串标准，自己构造了一个字符串类型。而在构建的字符串类型中加入len和free属性，len属性使Redis获取字符串值的长度的时间复杂度为O(1)、free属性作用域内存重分配，方便于字符串的修改。

整数与小数是看存储的value是否带小数，看如下示例

hash

list

C语言底层没有链表结构，Redis自己封装了一个链表结构，而该链表结构包含头部节点和尾部节点的指向，所以Redis的链表结构操作头部和操作尾部都是很方便的，也由此Redis的列表是支持栈和队列类似的功能的

set

无序集合，字符串成员，不允许重复，底层通过哈希表实现，添加、删除和查找时间复杂度都是O(1)，每个集合能存储2^23 - 1个成员。

zset

有续集合，字符串成员，每个成员关联一个double类型分数，成员不允许重复。跟set集合一样，底层都是使用哈希表实现的

需要注意有序集合默认按分数大小排序，对应相同的分数会根据成员名称的编码顺序排序。

如果不想使用默认的排序规则，应该设法保证分数不重复，或者做成即使重复也不影响。

想要分数不重复，可以处理一下分数，比如当分数确定是整数时可以给分数加上精确到秒的时间戳小数，取出后去掉小数部分即可。具体可根据业务处理

geo

该功能在Redis3.2版本后才支持

Redis数据库

Redis数据库

Redis默认创建16个数据库。数据库数量可以在redis.conf中配置

Redis服务器的不同数据库的数据是独立的，相当于一个MySQL服务器可以创建多个数据库。Redis数据库底层是通过哈希表来存储数据，可以简单理解成每个数据库就是一个哈希表结构，每个key-value的key都是字符串，value则可以是各种Redis数据类型。

Redis客户端的目标数据库dbid默认是0，但是每个Redis客户端可以使用select指令切换到不同的数据库（这里说的只是单机Redis服务器下，集群模式只有一个db0），不过不建议客户端这么做，我个人建议使用默认的0号数据库。

其他15个数据库那还有用吗，我目前用的比较少，如果说一定要找个用法的话，我觉得在机器资源比较少，并且只有一个Redis进程的话，可以考虑用不同dbid表示的数据库存储不同环境的缓存数据。

过期键清除策略

Redis使用惰性删除和定期删除两种策略，两种方式配合服务器可以更好地在使用CPU时间和避免浪费内存空间之间取得平衡

  • 惰性删除：放任key过期不管，但是每次操作这个key都检查一下是否过期，如果过期就删除
  • 定期删除：每隔一段时间程序就检查一下数据库，删除里面的过期key

RDB持久化策略不会备份过期键，恢复的时候已过期的键也会被忽略不会载入

AOF持久化策略只要key没有被删除就会做备份，当key被删除后，程序会向AOF文件追加del指令删除该key。重写的时候如果key被删除了，就会移除掉。

持久化

Redis是内存数据库，安全性很重要。持久化是解决数据安全问题的其中一个方式。Redis提供了RDB和AOF两种持久化方式，各有不同优缺点，当然其中一个方式的缺点就是另一个方式的优点，使用者可自行选择。

持久化机制

RDB备份方式每次备份都是全量备份，备份过程会生成.rdb文件，备份时生成新的文件，备份完成后移除旧的备份文件，以此保证最新状态都保存在dump.rdb。

也正因为每次备份都是全量备份，当前内存中有多少数据就保存多少数据，所以有很多种方式可以优化备份存储工作，比如使用二进制压缩存储，这种压缩存储方式使得这种备份后的文件相对AOF备份的文件小的多

另外这种备份方式文件存储的都是数据，Redis启动恢复的时候只需要把数据加载到内存中即可，这也是RDB备份方法相对AOF备份方法恢复更快的原因

在RDB备份的时候，Redis服务器可以选择fork一个进程来操作持久化备份工作（bgsave指令），期间不影响服务的正常运行。

Redis有两个指令可以触发该持久化备份，save指令和bgsave指令，两个指令不同之处

  • save：立即备份，服务端主进程执行备份，因为Redis是单线程的，所以备份期间不能处理客户端请求，慎用该指令，特别是数据量大的生产环境
  • bgsave：fork子进程执行备份，服务端（父进程）可以正常处理客户端请求

bgsave指令执行过程中发起的save和bgsave指令都会被拒绝，后面AOF持久化会用到bgrewriteaof指令，三个指令并发关系是

  • bgsave指令执行期间，发起的save指令会被拒绝，发起的bgrewriteaof指令延迟到bgsave完成后执行
  • bgrewriteaof指令执行期间，发起的save和bgsave指令都会被拒绝
    • 这是出于性能考虑，同时fork多个进程会影响Redis服务端的性能

RDB持久化机制是服务端每个一段时间全量备份一次，内部使用的就是bgsave指令，可以通过redis.conf文件配置触发备份时间，默认的触发机制如下

Redis启动加载RDB备份是自动完成的，无需人工做任何操作。比如正常我们启动Redis会打印对应的日志：

在Redis服务初始化之后会加载备份数据，还原Redis状态，等一切准备就绪，服务端才开始接受客户端连接和请求

优缺点

  • 优点
    • 每次备份生成dump.rdb文件，这个文件灾备简单，直接copy到其他地方就可用
    • 节省存储空间
    • 恢复性能较快
  • 缺点
    • 全量备份导致备份触发机制只能是定时备份，如果备份频率高，每次全量备份会消耗性能
    • 定时备份除了做不到实时，当服务端宕机会有一小部分数据未备份
    • 当数据量大的时候，每次全量备份时期服务端性能会下降：CPU和内存激增

AOF持久化

AOF则需要用户手动启用，这种备份方式是通过备份客户端写指令的方式做到实时或者相对实时效果的，所以磁盘的写入操作是追加写入，不需要修改已写入数据，对应文件是*.aof文件，文件名也是可以配置的，默认是appendonly.aof

这种备份方式会有一个问题就是同一个key频繁操作，备份文件也会保存很多个这个key的指令，为了解决这个问题，Redis使用了重写机制，重写机制就是当.aof文件大小超过某些条件的时候触发重写，而所谓重写就是根据内存中数据的状态生成新的操作指令。比如原来有个字符串key，前后操作了100次写入，aof文件也对应着100条写指令备份，重写后就只有一条指令: set key value。触发重写必须满足两个条件，对应redis.conf的配置项为
