RabbitMq系列二之Amqplib库(上)

发表于 2018-09-15 | 分类于 JavaScript ， NodeJs

发现写博客有一个好处，它能催促你去了解更多，做更多的总结。前一篇文章介绍了 MQ，不过关于 MQ 的技术细节都没有提到。对于我自己来说，接触一个新技术的时候，一般是先写个 demo 玩起来先，然后再去深入接触和了解这个技术。本文会用到 Node.js 的 Amqplib 库，Node.js 会使用它来操作 RabbitMq，这也是RabbitMq官方推荐我们使用的库

安装 Amqplib

交互过程讲解

为了形象地讲述Amqplib扮演的角色，我们来做一个不恰当的类比。写过node.js的人多多少少了解MongoDB，也多多少少会用到mongoose。我们知道mongoose是MongoDB的ORM，使用mongoose能够很方便地操作MongoDB。

如果我们将RabbitMq类比成MongoDB，那Amqplib就是RabbitMq的ORM了。这个类比不是很恰当，RabbitMq不是数据库，它没有数据库和数据库表的概念，那Amqplib也就不是ORM了。举这个不恰当的例子只是加快我们知道Amqplib在这个过程中扮演什么角色而已。

下面的时序图包含了生产者、RabbitMq和消费者之间的交互过程：

图中，分别有三个角色：生成者，消费者，RabbitMq（为了更加直观，将RabbitMq的交换器和队列展现出来），从上往下分别是交互的时间顺序。

整体一看，扫一下时间轴，我们发现消费者在生产者发布消息前就已经在监听消息队列了，简单地说是先有了消费者，再有生产者。其实先有生产者还是先有消费者，没有对与错之分，我的习惯是先有消费者，这也是我以后会坚持的。先有了消费者能够避免消费者迟迟未上线消费消息导致消息积压，而RabbitMq的消息是存储在内存中的，多多少少会存在一些隐患。

另外生产者和消费者都可以声明队列，关于谁来声明队列的问题，RabbitMq是建议生产者和消费者都应该尝试着创建队列。但是有时业务本事在架构之初已经充分预估了队列的使用情况，完全可以在业务程序上线之前创建队列，这样业务程序可以免去声明过程，直接可以使用。RabbitMq对于重复声明队列的请求是友好的，每一次客户端请求声明队列的时候，RabbitMq会先判断队列存不存在，如果不存在就创建队列并成功返回，如果存在，则不创建队列，同时也成功返回。

连接RabbitMq

客户端需要先与RabbitMq建立连接才能执行之后的操作，这个很容易理解，网络通信的第一步是建立连接。与RabbitMq建立连接很简单：

连接RabbitMq的url格式为：amqp://用户名:密码@host:5672，这里首先连接RabbitMq并在一秒后关闭连接。连接上RabbitMq之后，从RabbitMq的管理后台的概述(overview)菜单 -> 汇总(totals) -> 全局统计(global counts) -> 连接数(connections) 可以看到：

另外点击连接菜单(connections)进去也能够看到连接详情，包括连接的状态，信道数量等等：

通过amqplib.connect，客户端与RabbitMq建立了一个TCP连接，一旦成功建立了TCP连接，客户端就可以轻松的操作RabbitMq，包括创建信道、收发消息等等一切操作。

RabbitMq不建议生产者每次发送消息的时候都建立一个TCP连接，而是公用一个或者几个TCP连接。我建议我们在项目启动之初就创建这个TCP连接，之后的操作都是基于这个TCP连接。因为RabbitMq的连接数是有限制的，当超过这个限制，就会报错:

所以我们应该建立公用的connection，毕竟我们都不想辛苦写出来的程序因为连接数超限制而报错。

创建信道

RabbitMq的所有操作都是基于信道完成的，这是因为TCP连接的资源有限，信道不限制。当我们与RabbitMq建立连接之后，我们可以基于TCP连接创建信道，表示我们打开了一条与RabbitMq通信的私密通道，基于这条路径我们就可以使用操作RabbitMq了。下面是TCP与信道的关系图：

我们可以把TCP连接当成一条电缆，信道就相当于电缆中的光纤束。一般来说我们一个应用程序创建一条TCP连接，在这条连接上创建多条信道。

声明队列

队列是AMQP中高level层面的模块，客户端需要确保在使用它之前创建队列。生产者和消费者都可以尝试声明队列，RabbitMq对于队列的重复声明采取的做法是什么都不做并成功返回，但是这个限于声明的时候options参数要一致，如果不一致将会报错。

声明交换器

交换器跟队列一样，是高level层面的模块。网上搜RabbitMq的交换器，基本都会涉及到消息是如何到达队列的？的问题，这是因为交换器的作用就是路由生产者发送的消息到指定的队列。

交换器的声明和队列的声明类似，客户端在使用它之前确保已经成功创建了它，重复声明当参数不一致的时候会报错。

关于路由键和绑定键，下面的 队列绑定交换器 小节会讲到。

简单来说，当交换器和队列声明完毕之后，我们需要将队列绑定打交换器，这样当生产者将消息发送给交换器的时候交换器可以知道要把消息给哪条队列了。

大致流程是：

  • 队列和交换器绑定，并在绑定的时候提供一个字符串的绑定键 bindKey
  • 生成者发送消息的时候提供一个字符串的路由键 routingKey
  • 交换器根据交换器的类型以及 bindKey 和 routingKey 的匹配关系来决定将消息路由给哪条队列
  • 一旦交换器匹配不到队列，消息就会进入黑洞。也就是消息投送给一个不存在的队列，客户端任然是可以收到消息投递成功的返回结果，只是这条消息不进入任何一条队列，它最终会消失。这就相当于往天上打了一枪，子弹是飞出去了，但是最终不知道它去了哪里。

为了能满足各种消息投递需求，RabbitMq提供了四种交换器：direct，fanout，topic和headers

topic - 模糊匹配 (where routingKey like bindKey)

在这个过程中，队列绑定到交换器提供一个模糊匹配的字符串绑定键，当消息的路由键符合绑定键的匹配规则的时候，消息被路由到该队列。

绑定键有三个字符：.、*、#是有特殊含义的

  • .字符将绑定键分成了多个部分，如绑定键为 log.error 时有两个部分

  • *字符匹配某个部分的内容，如 log.* 将匹配 log.任意字符，但是不匹配 log.任意字符.aaa

  • #字符匹配多个部分的内容，如log.#可以匹配log、log.任意字符、log.任意字符.任意字符...

headers - 现在很少用到

队列绑定交换器

绑定交换器的部分在前一部分已经讲了很多，关于绑定的概念也有了些许了解。但是当我们刚安装启动RabbitMq的时候，我们从management界面上的exchanges导航下可以看到，其实RabbitMq已经默认为我们提供了一些交换器，平时我们不一定每次都需要自己声明新的交换器，拿已经存在的交换器就好了。

首先我们可以看到，这些交换器都是持久化的(在features里面有个青色的D)，这里包含了2个direct类型、2个headers类型、1个fanout类型以及3个topic类型的交换器

第一个direct类型的交换器是RabbitMq默认的交换器，其他的交换器也只是RabbitMq提供好方便我们使用而已

默认交换器点进去不会列出已经绑定的队列列表，它是所以队列默认会绑定的交换器。也就是每个队列声明成功之后都会绑定到这个交换器，绑定键就是队列名称，客户端不能显示使用这个默认交换器绑定或者解绑队列，这个交换器也不能删除。

默认交换器如何使用：

当我们使用sendToQueue发送消息给某个队列的时候，默认交换器就用上了。消息到达默认交换器，默认交换器根据队列名字将消息路由到指定队列

所以我们可以发现，所有的消息都是经过了交换器，最终才到达队列的
