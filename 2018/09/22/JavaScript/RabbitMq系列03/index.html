生产者发送消息

当交换器和队列已经声明完毕，我们就可以准备发送消息了。amqplib提供了publish和sendToQueue两个方法可以发送消息，两个方法中sendToQueue简单的将消息发送给指定的队列(经过默认交换器)，publish则是根据路由键将消息发送到指定的交换器。两个方法表面形式上有所区别，但是消息路由过程以及参数都是一样的。

sendToQueue(queue, msg, options)

发送过程很简单，queue参数是队列名称，msg是消息体的buffer对象，options参数是消息的设置参数

options常用参数：

  • expiration (字符串 | 数字)

消息超时时间，一旦消息超过某个时间没有被消费就会被删除。规范中，这是一个字符串；这里提供的数字将被强制转换为字符串。

  • priority (正整数)

消息的优先级。

  • deliveryMod (布尔型或数值型)

或者 1 / false，意思是非持久的; 或者 2 / true，意思是持久的

  • replyT (字符串)

通常用于在RPC方案中命名接收应用程序必须向其发送回复的队列

publish(exchange, routingKey, msg, options)

前三个参数很容易知道，exchange参数是交换器名称，routingKey参数是路由键，msg参数是消息体的Buffer，options参数跟sendToQueue方法是一样的

注意：消息可以使用deliveryMod参数设置为持久化，也就是即使RabbitMq重启，消息也不会丢失。但是这个功能依赖于队列的持久化，也就是要想消息持久化，队列声明的时候也应该设置为持久化。这个也很容易理解，队列如果不是持久化的，RabbitMq重启，队列都消失了，消息自然也就丢失了

生产者发送确认模式

RabbitMq的消息传递代理的系统是分布式的，发送方不能保证消息发送成功甚至被对方成功处理(针对能否被成功处理RabbitMq支持了RPC模式)，因此生产者和消费者都需要一种交互和处理确认的机制。

那生产者怎么知道消息已经成功发送了呢，amqplib提供了confirmChannel来实现这个功能

我们尝试发送10万条消息到队列，但是我们选择了一个不存在的队列。我们发现消息还是正常发送并且被确认了。看起来像是消息确认不真正确保消息一定是入了队列，即使消息进入黑洞，也表示正常发送。文档的说明方法就是判断发送回调的err是否为null，为null表示消息成功，然后确认打印消息已经确认。

但是我发送这10条消息需要20多秒钟的时间，等20多秒钟结束之后(消息全部发送给了RabbitMq)，才执行All messages done

消费者监听队列，消费消息

消费者有两个形式可以消费队列的消息：推模式 和 拉模式。堆模式采用 Basic.Consume 进行消费，拉模式采用 Basic.Get 进行消费

推模式

推模式，消费者监听队列，每当队列有消息就会推送给消费者消费。消费者需要做的是监听，写好消费消息的逻辑

options参数:

  • consumerTag (string)

RabbitMq 用这个名称参数区分消息传递的消费者，他不能再信道中使用。这个经常被忽略，这时 RabbitMq 会创建一个随机的名称

  • noAck (boolean)

消息确认方式，如果为 true，当消息发送给消费者之后 RabbitMq 就会将它从队列中移除，而不会等待消费者处理完后告诉 RabbitMq。默认是 false

很明显如果设置为true，是可以提高 RabbitMq 性能的，毕竟 RabbitMq 只管将消息轮询 (消息按数量平摊给消费者们，如当前有6条消息和两个消费者，1、3、5条消息会发给先监听队列的消费者；2、4、6条消息发给后监听的消费者) 地发送出去

很多情况下我们不能将它设置为 true。首先消息发送给消费者之后自动出队，如果这个时候消费者窘机了，消息也就找不回了；然后设置为 true，prefetch 并发就没效果了 (下面的消息确认会讲到这个)

  • priority (integer)

给予消费者优先权; 较高优先级的消费者优先于较低优先级的消费者获得消息

拉模式

拉模式，消费者主动从队列请求一个消息

正常情况下，当队列queue里面存在待消费的消息的时候能够正常得到消息(如下图)，但是当队列不存在待消费的消息，返回false

当队列queue不存在的时候，会报错

这个拉模式的options参数只有一个参数: noAck，含义与推模式的 noAck 一样

消息确认

RabbitMq 怎么决定一个消息已经被正常处理了呢？这就是消息声明周期的最后一步：消息确认。消息一旦被确认，RabbitMq就会知道这个消息已经被正常处理，然后将消息从队列移除。

前面的推拉模式，我们都显示添加了options参数： {noAck: true}，这是最简单的消息确认模式，即消息发送即确认。当消息发送给消费者之后，消息就从队列中移除，不管消费者如何消费，顺利处理还是报错退出，消息不会复存。很多时候我们会觉得这种消息确认方式是不安全的，我们经常更希望消息能够正常处理后，消息才结束并从队列移除。

如果我们想要手动确认消息，我们在消费消息的时候就需要将noAck参数设置为false，当然这个参数默认就是false，我们不显示设置这个参数默认就是需要手动确认消息。

没有显示设置noAck为true，默认需要手动确认消息，但是这段代码我们没有显示手动确认消息，当我们运行的时候会发现消息已经从ready变成了unacked状态，表示消息等待确认。

一个待消费的消息状态为Ready，当消费者不管使用推模式还是拉模式使用{noAck: false}得到一个消息时，在手动确认之前，状态会变为unacked状态。这个时候如果这个消费者一直处于这个状态不确认消息，也不断开连接，这个消息会一直保持这个状态，其他消费者也不能消费这个消息。这个时候这个消费者有两种选择处理这个消息，一种是手动确认消息，消息最终从队列移除；另外一种是关闭连接，消息变回Ready状态，等待下一个消费者消费该消息

手动确认消息很简单，只需要在消息处理完毕之后调用一个方法即可: channel.ack(msg)

消息分发

当我们有多个消费者在消费统一个队列的时候，消息是怎么分配给众多消费者的？

首先需要说明的一点就是，RabbitMq的一条队列的一个消息只会发送给一个消费者处理，永远不会将同一个消息发送给多个消费者同时处理。只有一种情况一条消息前后经历过多个消费者处理，那就是前面处理的消费者没有确认消息，在它断开连接之后这条消息重新变成待消费状态之后发送给另外一个消费者处理。RabbitMq拒绝这种会导致功能变得复杂且难以理解的功能，如果我们期待一条消息可以被多个消费者处理，那我们应该声明一个fanout交换器，每个需要处理消息的消费者都将队列绑定到这个交换器，那么当有消息就可以收到了。

Round-robin（轮询分发）

默认这种方式，RabbitMq会将消息分摊给每个消费者。如果当前队列有10条消息，有两个消费者，则[1, 3, 5, 7, 9]条消息将推送给先监听队列的消费者，[2, 4, 6, 8, 10]条消息将推送给后监听队列的消费者

但是有时候我们不希望使用默认的这种方式，毕竟两个消费者处理能力不一样，处理消息能力强的消费者可能很快处理完了5条消息，处理能力弱的能需要很长的时间来处理一条消息。这个时候我们就希望RabbitMq按照处理能力分发消息，这就是下面讲到的另一个分发规则

Fair dispatch（公平分发）

这种方式，消费者消息的时候不能将 noAck 参数设置为true，消息确认方式应该保证为手动确认

这种方式，我们使用basicQos( prefetchCount)方法，来限制RabbitMQ只发不超过指定条数的消息给同一个消费者。当消息处理完毕后，有了反馈，才会进行第二次发送。

下面的逻辑将prefetch设置为1，表示消费者处理消息的并发值为1。RabbitMq会给消费者发送1条消息，在消息被确认之前不会发送第二条消息，而一旦消息被确认，马上发送一条新的消息，以保证该消费者始终有一条处理中的消息。

代码的不同之处在于加了代码：channel.prefetch(1)。这行代码的位置没有限制，它对信道做了限制，一般在channel.consume之前指定就可以了。

如果不控制并发prefetch，或者noAck为true，RabbitMq会一次性将所有消息分摊给消费者，这经常会出现问题，如果两个消费者，队列有一百条消息，则每个消费者各得50条消息，相当于并发50。这种方法对服务不利，我们不应该出现这种情况，应该控制并发保证服务高可用。
